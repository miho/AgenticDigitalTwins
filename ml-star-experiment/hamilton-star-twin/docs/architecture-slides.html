Hamilton STAR
Digital Twin

A firmware-faithful, state-machine-driven instrument simulator with a shared datamodel between simulation, visualization, and AI-agent introspection.
236 FW commands  ·  10 SCXML modules  ·  33 step types  ·  439 passing tests
THE BIG IDEA

Three decisions.

01

Firmware is the contract.

The boundary of the simulation is the Hamilton C0 firmware protocol — same ASCII commands the physical instrument's master controller accepts. VENUS, the TCP bridge, raw MCP calls, and in-memory tests all cross the same line.

02

SCXML is the logic.

Each hardware module is a W3C-compliant state chart. Transitions are data, authored in VSCXML, compiled to JS. There is no imperative engine pretending to be a state machine.

03

The datamodel is shared.

Positions, tip state, volumes, liquid components — one store. The renderer reads it, physics observes it, the HTTP API snapshots it, the MCP agent queries it. No DTOs. No drift.

Why not the usual approaches

ApproachWhere it livesWhy it falls short
Protocol-level simulator VENUS step layer Each step is a black box; can't catch FW-level failures; the real VENUS runtime won't drive it.
Full-stack emulator C/C++ binary that mimics FW Heavy, opaque, non-portable; physics is baked into the same code that implements the protocol.
Paper & Excel Human-written procedures Doesn't execute; doesn't catch collisions or dead-volume underflow; no feedback loop for LLMs.
GOAL
A simulator that speaks firmware — whatever drives the real instrument drives the twin. Visualization, physics, audits, and agent tool-use all derive from that one execution path.

PRINCIPLE 01

Firmware as the contract

What a command actually looks like

C0ASid0004xp02755yp05300av01000tm01lm0zp01800

    │    │        │        │   │   │    └──  Z target (18 mm)
    │    │        │        │   │   └───────  LLD mode

WHY IT MATTERS
VENUS drives the twin bit-for-bit unchanged. It opens a TCP connection to the bridge, sends exactly this string, and reads back an ACK identical to what a real STAR would return.

Every client, one execution path

VENUS

Opens TCP to :34567. Sends FW strings. Gets ACKs. Doesn't know it isn't talking to a physical STAR.

Browser

Buttons + protocol editor synthesize FW strings, POST /command. SSE stream returns motion envelopes + assessments.

LLM agent

MCP twin.sendCommand { raw: "C0AS…" }. Same string space. Plus docs.* tools to discover what's available.

Tests

Hold a DigitalTwin in-memory, call sendCommand synchronously. Same pipeline as production.

PROVABLE FIDELITY
fw-trace-replay.test.ts replays real VENUS traces, asserts byte-identical ACKs.
PRINCIPLE 02

SCXML as module logic

Hardware modules are state machines.

  • A PIP channel is no_tip → tip_empty → tip_loaded. It cannot dispense from tip_empty — the firmware refuses.
  • iSWAP: idle → gripping → holding → moving → releasing. A misordered command drops into error and latches.
  • Master owns the sys-init sequence — 81 commands, most only legal after the appropriate init step.
THE IMPERATIVE TRAP
Writing this as if (state === X && !flag && !errorPending) produces a tangle. Writing it as SCXML produces a picture that is the program.

PIP channel state chart (simplified)

stateDiagram-v2 [*] --> no_tip no_tip --> tip_empty : C0TP tip_empty --> no_tip : C0TR tip_empty --> tip_loaded : C0AS tip_loaded --> tip_empty : C0DS (dm=4, blowout) tip_loaded --> tip_loaded : C0AS tip_loaded --> tip_loaded : C0DS (dm=2/3) tip_loaded --> no_tip : C0TR tip_empty --> error : C0AS (tm conflict) tip_loaded --> error : C0TP (already loaded)

A transition's effect

if (this._matchEvent(event.name, 'C0AS')) {
  this._execScript(`
    var av = _event.data.av || 0;
    var total = 0;
    for (var i = 0; i < 16; i++) {
      if (tip_fitted[i]) { volume[i] = av; total += av; }
    }
    active_volume_total = total;
    if (_event.data.xp !== undefined) pos_x = _event.data.xp;
    if (_event.data.yp !== undefined) { /* per-channel Y */ }
    if (_event.data.zp !== undefined) {
      for (var k = 0; k < 16; k++) {
        if (tip_fitted[k]) pos_z[k] = _event.data.zp;
      }
    }
  `);
}

This inline script is the only place these fields are written for this event. No "backing store". No mutation method. The datamodel assignments are the transition's effect.

PRINCIPLE 03

One shared datamodel

The trap most simulators fall into

[ UI state ]  ←  custom sync code  ←  [ engine state ]
                                             ↑
                                [ separate physics state ]
                                             ↑
                                [ tests mock their own state ]
EVERY ARROW IS A BUG
Three stores drift. Tests pass against mocks that don't match production. The UI shows 100 µL when the engine knows it's 50. Users debug ghosts.

What the twin does instead

One store. One writer per field. Every reader looks at the same thing.

Concrete payoff

USER REPORT
"The well shows 100 µL in the tooltip but I aspirated 200 µL and got no warning."
  • One place to look. Not three stores. Not "which cache is stale". The shared datamodel either has the right value or doesn't.
  • Tests check production state. After C0AS, a unit test reads the same wellVolumes map the browser renders from.
  • VENUS and the browser always agree. When VENUS polls state, it sees the same numbers a developer's monitor is rendering in real time.
PRINCIPLE 04

Motion is real

Physical time, not cosmetic animation

A real C0AS descends Z, runs the plunger, retracts — ~2 s for a typical 100 µL aspirate. The twin represents that time explicitly.

interface MotionEnvelope {
  arm: "pip" | "iswap" | "h96" | "h384" | "autoload";
  startX: number; startY: number;
  endX: number;   endY: number;
  startZ?: number; endZ?: number;
  dwellZ?: number;           // Z-bob: descend → hold → retract
  startRotation?: number;    endRotation?: number;
  startGripWidth?: number;   endGripWidth?: number;
  startTime: number;         // Date.now() at emit
  durationMs: number;        // physics-plugin-computed
  command: string;
}

Every temporal thing in the twin derives from this envelope.

PRINCIPLE 05

Physics is observational

Two-phase plugin contract

interface PhysicsPlugin {
  // Pre-commit: can REJECT (errorCode). Hard safety invariants.
  validateCommand?(event, data, deckTracker, dm):
    { valid: boolean; errorCode?: number };

  // Pre-commit: stamps `_delay`, computes derived fields. Never rejects.
  onBeforeEvent?(event, data): data;

  // Post-commit: OBSERVES. Never rejects. Emits AssessmentEvents.
  assess?(event, data, deckTracker): AssessmentEvent[];
}

Validation REJECTS

Tip mask conflict · Z below traverse while X moving · tip already fitted. Error code returned; nothing else happens.

Assessment REPORTS

empty_aspiration · volume_underflow · contamination · LLD-no-liquid · TADM-deviation · collision · air_in_dispense.

Why the split matters

CONFLATING REJECT + REPORT
Produces simulators that either refuse too much ("too fussy to run real methods") or refuse too little ("method ran but dispensed air into three wells silently").

CAPABILITIES

What this unlocks

Agent-driven automation

An LLM with MCP access runs a method, observes assessments, adapts — on the simulator first, then on real hardware. Same FW boundary for both.

Reproducibility

Every trace replayable offline. A failed real-instrument run snapshot-shares + re-opens anywhere a twin runs.

What-if training

Fork a trace at any command, propose an alternative, diff the outcome. For teaching, post-mortems, pre-flight checks.

Zero-ceremony introspection

Any developer, operator, or agent asks the twin what it can do. The twin answers from the same codepath the execution lives in.
