# Test Audit — 2026-04-16

**Scope:** `tests/integration/*.test.ts` (195 tests across 10 files) + `tests/e2e/visual.test.ts` (46 tests).
**Purpose:** Identify tests that pass while hiding real errors, per the criteria in `tests/TESTING-GUIDE.md`. This audit drives Phase 0 remediation (task #8).
**Methodology:** Each test file was reviewed for the banned patterns from TESTING-GUIDE.md:
1. `accepted: true` as the sole/primary assertion (no physical outcome check)
2. `errorCode.toBeGreaterThan(0)` — doesn't pin the expected error code
3. `toBeDefined()` without a subsequent value assertion
4. Derived-value checks (e.g., `pip.volume[0]` after aspirate) without matching source-volume check
5. `length.toBeGreaterThan(0)` on arrays without content verification

## Classification

- **Strong**: Every mutation is bounded by capture-before/capture-after; error paths pin specific error codes; `toBeDefined()` paired with value/shape assertions.
- **Medium**: Generally solid but has at least one weak pattern that should be strengthened.
- **Weak**: The test could pass while the twin is broken. Must be strengthened in task #8.

## Top 23 weak tests (for Phase 0 remediation — task #8)

Ordered by priority (highest first = biggest risk of hiding errors).

### Priority 1 — Error codes unpinned

These tests pass for ANY non-zero error code, which means a bug that changes error 8 to error 99 would not be caught.

1. **tutorial-workflow.test.ts:128-133** — `"rejects aspirate without tips"`
   - Current: `expect(result.errorCode).toBeGreaterThan(0)`
   - Fix: `expect(result.errorCode).toBe(8)` (error 8 = "no tip fitted")
   - Why: Any rejection currently passes this test. A regression that changes the rejection reason would slip through.

2. **foundation-fixes.test.ts:73-77** — `"TCC rejects temperature above 105C"`
   - Current: `expect(r.errorCode).toBeGreaterThan(0)`
   - Fix: `expect(r.errorCode).toBe(19)` (error 19 = temperature error)
   - Also: verify `r.errorDescription` contains "temperature" or "overtemp".

3. **tutorial-workflow.test.ts: similar rejects test for double tip pickup** — verify error 7 specifically.

4. **ghost-commands.test.ts: two `errorCode > 0` occurrences** — each must pin the expected code.

### Priority 2 — Missing source-volume checks

Channel volumes are derived state — they can be wrong even if set correctly (e.g., twin adds volume without actually tracking source depletion).

   - Fix: capture `getColumnVolumes` before, verify decrease after matches aspirate volume
   - Why: The twin could set pip.volume without decrementing sources — test wouldn't notice.

6. **tutorial-workflow.test.ts:172+** — `"generates TADM assessment event"`
   - Current: checks TADM event fired

7. **foundation-fixes.test.ts:84-105** — `"tip type 4 uses 10000 correction entry"`

## Running the audit checks

The quantitative pass data was gathered by:

```bash
for f in tests/integration/*.test.ts; do
  name=$(basename "$f")
  total=$(grep -cE "^\s*it\(|^\s*test\(" "$f")
  acceptedOnly=$(grep -c "expect.*accepted.*toBe(true)" "$f")
  errorCodeGt=$(grep -c "errorCode.*toBeGreaterThan(0)" "$f")
  toBeDef=$(grep -c "toBeDefined" "$f")
  echo "$name: total=$total accepted=$acceptedOnly errcodeGT=$errorCodeGt toBeDef=$toBeDef"
done
```

These metrics are not a perfect signal (an `accepted: true` check surrounded by state assertions is fine), but they flag files worth closer review.
